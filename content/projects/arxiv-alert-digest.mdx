## 为什么做

我订阅了 arXiv 的 daily alert，但一封邮件常常塞进几百到一千多篇论文标题、作者、分类和摘要。直接从头读到尾成本太高，而且很多论文和我的研究方向关系不大。

我真正需要的是一个研究助理式的筛选器：

- 先把邮件里的所有论文结构化解析出来
- 再判断它们和我的研究方向有多相关
- 最后只留下值得看的论文、方法灵感和跨领域迁移点

这个工具就是为了解决这个每天都会出现的文献过载问题。

## 做了什么

它会自动连接 QQ 邮箱，拉取来自 arXiv.org 的 daily 邮件，解析出每篇论文的 arXiv ID、标题、作者、分类、摘要和链接，然后调用 DeepSeek API 分批进行科研语义筛选。

**处理流水线**

```text
QQ 邮箱 IMAP
  -> 搜索 arXiv daily alert
  -> 解析邮件正文论文块
  -> DeepSeek 批量判断相关度
  -> 生成个性化总览
  -> 输出 Markdown 文献摘要
```

**相关度分层**

| 等级 | 含义 |
|---|---|
| HIGH | 方法、数据或发现可以直接进入我的研究 |
| MEDIUM | 同一大领域，方法可借鉴或适配 |
| LOW | 边缘相关，可能提供灵感 |
| NONE | 暂时无明显关联 |

## 我的研究方向配置

项目把研究方向写进 `config.yaml`，LLM 不只是翻译摘要，而是带着明确研究上下文做判断。

当前配置聚焦：

- 城市更新与时空数据挖掘
- LLM 和多智能体系统用于城市低效用地识别
- GEE、LandTrendr、遥感变化检测与施工审批数据融合
- 多源数据融合、语义匹配和城市场景智能体应用

这样筛出来的结果不是“今天 arXiv 有什么”，而是“今天哪些论文可能真的能推进我的课题”。

## 一次真实输出

某次 arXiv daily 邮件里解析出 942 篇论文，系统按研究方向完成了相关度判断：

| 结果 | 数量 |
|---|---:|
| HIGH 直接相关 | 31 |
| MEDIUM 可借鉴 | 221 |
| LOW 可能有灵感 | 547 |
| NONE 暂不相关 | 143 |

输出的 Markdown 会先给出个性化总览，再按关联度列出论文表格：

```markdown
# arXiv Daily Digest — 2026-06-03

> 共 942 篇 | HIGH 31 | MEDIUM 221 | LOW 547 | NONE 143 | 显示 799 篇

## 个性化总览

本期与研究方向高度相关的进展主要体现在：
多智能体系统协作效率、安全性与可靠性；
遥感与地理空间 AI 的交叉融合；
LLM 与结构化知识、知识图谱、因果图的结合。

## 相关论文

| # | 论文 | 分类 | 一句话总结 |
|---|------|------|------------|
| 1 | Plan2Map | cs.CV cs.AI | 从规划文档重建地理空间边界，可直接对应施工审批数据解析。 |
| 2 | FAF-CD | cs.CV | 面向异源遥感影像的频率感知变化检测，可用于城市更新区域识别。 |
```

## 细节设计

**分批调用 LLM**

arXiv alert 邮件太长，直接一次性塞给模型不稳定。工具按 `batch_size` 分批处理论文，每批生成关联度和一句话总结，再汇总生成总览。

**断点续跑**

处理上千篇论文时，API 调用中断很常见。每完成一个批次都会保存 checkpoint，重跑时自动跳过已完成批次。

**去重处理**

每封邮件使用 Message-ID 记录处理状态，避免重复处理同一封 arXiv alert。

**最低关联度筛选**

可以在配置里设置 `min_relevance`，例如只看 HIGH / MEDIUM，也可以保留 LOW 用来寻找跨领域方法灵感。

## 日常使用

```bash
# 处理最近一封 arXiv 邮件
python main.py

# 处理最近 3 封
python main.py --emails 3

# 跳过最新一封，处理上一封
python main.py --pick 1

# 只拉取和解析邮件，不调用 LLM
python main.py --dry-run
```
